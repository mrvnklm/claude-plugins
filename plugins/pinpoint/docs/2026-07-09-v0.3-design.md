# pinpoint v0.3 — design & implementation spec

Status: approved 2026-07-09. Implements four changes on top of v0.2 (commit `fa09ad1`).

## Context

pinpoint is a Claude Code **Channel plugin**: a browser overlay lets you click UI
elements in your running dev app, type a note, and it streams into the Claude
session as `<channel source="pinpoint">` tags (screenshot + selector + url +
source hint). Status flows back via an `update_status` MCP tool over SSE.

Source files (edit these; the `bridge/*` files are **built artifacts** — do NOT
hand-edit them):

- `src/overlay.js` — the in-browser overlay (vanilla JS, Shadow DOM). Bundled by
  esbuild (`build.mjs`) → `bridge/overlay.dist.js`.
- `src/server.mjs` — the stdio MCP bridge + localhost HTTP listener. Copied/built
  → `bridge/server.mjs`.
- `build.mjs` — esbuild bundling. Run `node build.mjs` after editing `src/*`.
- `skills/pinpoint/SKILL.md` — usage doc.

Hard constraints:
- Overlay stays **vanilla JS inside a Shadow DOM**; no framework, no new runtime
  deps beyond the existing `@medv/finder` + `html-to-image`.
- Preserve all existing hardening (token gate, 24 MB body cap → 413, constant-time
  token compare, port fallback + persist, shutdown-with-session, screenshot id
  seeding, double-injection guard, `skipFonts:true`).
- Bump version to **0.3.0** in `package.json` and the MCP `Server({version})`.

## A) Live-delivery robustness (bridge) — the core bug fix

**Root cause:** `src/server.mjs` sends each task via
`mcp.notification({ method:'notifications/claude/channel', ... })` — fire-and-forget.
If no channel consumer is attached at that instant (plugin still loading, or Claude
launched without `--dangerously-load-development-channels`), the task is **lost** —
no queue, no ack, no re-delivery. This is the observed `#1 QUEUED, never received`.

**Design:**
1. **Disk-backed inbox.** On every accepted `/annotation` and `/followup`, append a
   record to `.pinpoint/inbox.jsonl`:
   `{ id, kind: "task"|"followup", content, meta, ts, delivered: false }`
   where `content` and `meta` are exactly what is passed to `mcp.notification`.
   - Keep an in-memory mirror loaded on boot (parse the file if present; tolerate a
     corrupt/partial trailing line — skip unpar-seable lines, never crash).
   - Bound the file: keep at most the last 200 records (rewrite on load if larger).
   - The inbox lives under `.pinpoint/` which is already git-ignored (`*`).
2. **New MCP tool `get_inbox`.** Declared in `ListTools`, handled in `CallTool`:
   - Input: `{ include_delivered?: boolean }` (default false).
   - Returns the pending (or all) records as a single text block using the SAME
     `content` format the channel tags use, each prefixed with its `id`/`kind`, so
     Claude can act on them identically. Include the screenshot paths already in
     `content`.
   - Side effect: mark returned pending records `delivered: true` and persist.
   - This is a **normal MCP tool call**, so it works even WITHOUT the channel flag —
     the reliable path.
3. **Keep the live push.** Still call `mcp.notification(...)` for instant delivery
   (happy path). The inbox is the safety net, not a replacement. On successful
   notification you MAY leave `delivered:false` — `get_inbox` is the explicit
   drain; do not auto-mark on notify (we can't confirm the peer consumed it).
4. **Update channel `instructions`** (the `instructions` string on the MCP
   `Server`): add a line telling Claude to call `get_inbox` at session start and
   whenever the user asks whether a task arrived, to pull anything queued while it
   was not connected. Also update `skills/pinpoint/SKILL.md` to mention `get_inbox`.

**Acceptance:** POST two annotations while nothing drains; `get_inbox` returns both
with full content + screenshot paths and marks them delivered; a second `get_inbox`
returns none. Corrupt inbox line does not crash boot. Existing `/health`,
`/overlay.js`, `/events`, `/annotation`, `/followup`, token gate, 413 cap all still
pass.

## B) Hover-through highlight fix (overlay)

**Root cause:** `overlay.js` `onMouseMove` → `elementUnderCursor(x,y)` sets
`host.style.pointerEvents='none'` so `elementFromPoint` sees *through* the panel.
Side effect: hovering **over the panel** returns the element *behind* it and
highlights it; `isOurs` never fires because the host was made invisible to hit-test.

**Design:** before the see-through resolution, do a normal `elementFromPoint`
(host interactive). If the topmost element `isOurs(...)` → `hideHighlight()` and
return early (we're hovering our own UI). Only when the topmost element is NOT ours
do the existing see-through pass. Apply the same guard in `onClickCapture` (it
already checks `composedPath().includes(host)`, which is correct — keep it, but the
early normal-hit-test also covers edge cases). No highlight while the cursor is over
panel/FAB/toast.

**Acceptance:** with pick mode ON, moving the cursor over the panel shows NO
highlight on elements behind it; moving over the page still highlights normally;
clicking an element still adds to cart; clicking panel buttons still works.

## C) Dockable sidebar (overlay)

**Root cause:** `.panel` is purely floating (`position:fixed; bottom:80px;
right:20px; width:340px`). No docked mode.

**Design:**
- Add a **dock toggle** in the panel header (icon button, e.g. ⧉ floating ⇄ ⇥
  docked), state persisted in `localStorage` key `pinpoint.dock` (`"float"` |
  `"dock"`), default `"float"`.
- **Docked** styling: `position:fixed; top:0; right:0; height:100vh; width:360px;
  border-radius:0; border-left:1px solid …; max-height:none`. The FAB stays
  (toggles pick mode); toast repositions so it isn't hidden under the sidebar.
- **Reflow (side-by-side).** On dock: set on `document.documentElement`
  `margin-right:<W>px` and `width:calc(100% - <W>px)`, and set a CSS custom property
  `--pinpoint-dock-width:<W>px` on `:root` so an app can optionally honor it for
  `position:fixed`/`100vw` elements (documented caveat: those otherwise ignore the
  html margin). On undock/close/panel-close: remove all three cleanly. Guard against
  leaving the page shifted if the overlay is torn down.
- Width: fixed 360px in v0.3 (no drag-resize — YAGNI). A vertical page-content
  reflow is automatic via the html width change.
- Keep floating behavior identical to today when in float mode.

**Acceptance:** toggling to docked pins a full-height right sidebar and the page
content reflows to end at its left edge; toggling back restores the page exactly
(no residual margin/width); choice survives reload; both modes usable for pick +
send + history.

## D) History / UI polish (overlay)

**Gaps today:** history row is badge + single-line truncated note + follow-up only;
`t.statusNote` from `update_status` is **saved but never rendered**; no time, count,
or thumbnail.

**Design (history rows):**
- Meta line: `#id · <relative time> · N Elem` (relative time from `t.ts`, e.g.
  "vor 3 min"; store `count` — already in the task object).
- Note shown on up to **2 lines** (clamp), not single-line ellipsis.
- **Render `statusNote`** under the row in muted text when present.
- **Small thumbnail:** store a **downscaled ~120px JPEG** of item 0 on the task
  object at send time (a few KB, not the full ~1 MB shot) and show it in the row.
  Respect the localStorage budget: cap history to 50 (already done) and thumbnails
  must stay small; if thumbnail generation fails, omit gracefully.
- Clearer status badges already exist — keep, ensure `blocked` is reachable.
- General cleanup of the cart/send area: consistent icons, clearer empty/disabled
  states. Keep the existing green accent (#10b981) system.

**Acceptance:** a sent task shows id + relative time + count + a small thumbnail;
after `update_status({status:"working", note:"…"})` the note appears live; history
survives reload; localStorage does not balloon (thumbnails are downscaled).

## F) Launcher as an edge tab / flag (overlay)

**Today:** the launcher is a floating round FAB fixed at `bottom:20px; right:20px`.

**Design:** replace it with an **edge-anchored tab ("flag")** — a vertical pill/tab
flush against a side edge, not a free-floating circle.
- **Position:** anchored to the left OR right edge, sitting lower by default (e.g.
  ~60% down the viewport height), not centered.
- **Draggable vertically:** drag it up/down along the edge; a plain click (no drag)
  still toggles the panel/pick mode. Distinguish click vs drag by a small movement
  threshold so a click isn't swallowed.
- **Side-snap:** dragging it across to the opposite edge snaps it to that side
  (left ⇄ right). The tab's shape/rounding flips to hug whichever edge it's on.
- **Persist** position + side in `localStorage` key `pinpoint.fab`
  (`{ side:"left"|"right", top:<number, px or vh fraction> }`), default e.g.
  `{ side:"left", top:0.6 }`. Clamp `top` so the tab stays fully on-screen on load
  and after viewport resize.
- Keep the active/pick state affordance (color/label change) and the
  `Cmd/Ctrl+Shift+K` shortcut.
- **Interaction with docked sidebar (C):** when the sidebar is docked on a side, the
  tab should not overlap it — prefer the tab on the opposite edge, or offset it clear
  of the sidebar. Keep it simple: if the tab's side == the docked side, nudge the tab
  to sit just inside/next to the sidebar edge (still reachable), or flip it. Pick the
  cleaner of the two during implementation and note the choice.

**Acceptance:** launcher renders as an edge tab (default lower-left); dragging moves
it vertically and snapping to the other edge changes sides; a click still opens the
panel; position/side survive reload; it never lands off-screen; docked sidebar and
tab don't visually collide.

## E) Rollout / versioning

- Bump `package.json` and MCP `Server` version → **0.3.0**.
- Run `node build.mjs`; verify `bridge/overlay.dist.js` and `bridge/server.mjs`
  regenerate and contain the new symbols (`get_inbox`, dock toggle, etc.).
- `claude plugin validate --strict` should still pass (if the CLI is available).
- Delivery of changes to a running setup:
  - Overlay (B/C/D): effective after a **page reload** (bridge serves fresh
    `overlay.js`).
  - Bridge (A): needs a **bridge restart** = new Claude session (the plugin MCP
    server re-spawns). One-time relaunch cost; after that delivery is robust.
- Do NOT touch anything outside `~/code/claude-plugins/plugins/pinpoint/`. The
  Tresor app (`~/vs-priv`) is a separate concern.
